# AGENTS.md — 開発ルール

このプロジェクトは、Next.js + TypeScript 製のブラウザ麻雀アプリです。
最終目標は「1人練習で強いAI」ではなく、**4人対戦でも世界一を本気で狙える最強麻雀AI**を作ることです。
コードを編集する自律エージェント (あなた) が守るべきルールをここに集約します。

## 最重要原則

1. **動く成果物を最優先する**。説明より実装。1人用練習画面と検証スクリプトが動かなくなる変更は禁止。
2. **AIの強さを最優先する**。見た目より、4人対戦で勝率・平均順位・放銃率・収支を改善するロジックを優先する。
3. **世界一を狙う前提で設計する**。局所ヒューリスティックで終わらせず、守備、押し引き、鳴き、点数、探索、自己対戦、学習へ接続できる形にする。
4. **後から拡張しやすい設計を保つ**。`lib/mahjong` と `lib/ai` を分離した状態を崩さない。
5. **動作確認は必須**。`npm run build` を通し、必要に応じて `npm test` / `npm run tournament` / `npm run dev` で確認する。

## 技術構成

- Next.js 14 App Router (`src/app`)
- TypeScript strict
- React 18
- 追加ランタイムは tsx (テスト・検証用) のみ。Node 標準 test runner を使う。
- 重い数値計算は将来 Python/WASM 実装に差し替え可能なよう、純関数を維持する。

## ディレクトリ規約

```
src/
  app/                    画面 (Next.js App Router)
  components/             UI部品
  lib/
    mahjong/              麻雀ルール、牌、シャンテン、受け入れ、和了、役、点数、4人対局
    ai/                   AI評価、推奨打牌、鳴き判断、リーチ判断、守備、探索、対戦エージェント
    game/                 1人練習用ゲームステート
scripts/                  自己対戦・4人対戦・シミュレーション用
python/                   学習・分析用 (今後)
test/                     Node test runner のテスト
```

新規ファイルは原則これらに収めること。`lib/ai` から `lib/mahjong` を import するのは可だが、逆は禁止。`lib/mahjong` から React に依存しないこと。

## エージェントの作業手順

1. 着手前に `PROJECT_STATUS.md`、`TODO.md` を読む
2. タスクを `TodoWrite` 等に分解する
3. AIの強さに効く変更を実装する
4. `npm run build` を必ず通す
5. 必要なら `npm test` (= `node --import tsx --test test/**/*.test.ts`) を通す
6. AI変更時は可能な限り `npm run selfplay` または `npm run tournament` で定量確認する
7. UI変更時は `npm run dev` を立ち上げ、ブラウザで挙動確認する
8. `PROJECT_STATUS.md` / `TODO.md` を更新する
9. Git commit する (commit メッセージは feat:/fix:/chore: のプレフィクス)
10. 余裕があれば `NEXT_PROMPT.md` を更新する

## 安全停止 (Safe Stop) ルール

以下のいずれかに該当する場合は、新規大変更を始めない:

- 利用量警告が見えた / 警告に近い気配がある
- 1つの大きな機能が完成した直後
- 大規模リファクタの前
- `npm run build` / `npm test` が通る安定状態に達した
- 作業が長引いて壊れそうなとき

安全停止時の作業:

1. `npm run build` / `npm test` を通す
2. AI変更があれば `npm run selfplay` または短い `npm run tournament -- 40 42` を通す
3. Git commit を作る (壊れていない状態をコミット)
4. `PROJECT_STATUS.md` を更新する
5. `TODO.md` を更新する
6. `NEXT_PROMPT.md` を「次のエージェントが貼ればそのまま再開できる」内容に更新する
7. 壊れている機能があれば `PROJECT_STATUS.md` に明記する

## 禁止事項

- `npm run build` が通らない状態でセッションを終える
- 大量の未検証変更を残したまま終える
- AI部分を後回しにし、UIだけを作り込む
- ユーザに細かい確認を求める
- 麻雀理論の長い解説をユーザに要求する
- 4人対戦で弱くなる変更を、ベンチマークなしに「改善」と呼ぶ

## AI 開発の優先順位

1. シャンテン数計算
2. 受け入れ枚数計算
3. 有効牌評価
4. ドラ・赤ドラ・役牌・タンヤオ・ピンフなどの打点評価
5. リーチ判断
6. 和了判定・役判定・最低限の点数計算
7. 危険牌推定 (現物 / スジ / 壁 / ワンチャンス / 河読み)
8. 攻撃/守備バランスと押し引き
9. 4人対局シミュレーションとAI同士の比較
10. 鳴き判断 (チー/ポン/カン、役牌ポン、喰いタン、守備副露)
11. テンパイ維持判断と待ちの質評価
12. モンテカルロ探索 / 期待値探索
13. 自己対戦ログの保存と統計分析
14. ランダムAI・単純シャンテンAI・攻撃AI・守備AIとの継続比較
15. Python による評価関数改善
16. 学習モデル化

ファイルを変えるたび、可能なら次のレベルを 1 つ上げる。特に `src/lib/ai` と `src/lib/mahjong` を触るときは、4人対戦の勝率・平均順位・放銃率のどれが改善するかを意識する。

## 既存実装のメンタルモデル

- 牌は `0..33` の整数で扱う (`src/lib/mahjong/types.ts` 参照)。
- `shantenAllFromCounts` が標準形/七対子/国士の最小値を返す。
- `evaluateDiscards` が 14 枚から各打牌候補を評価し、スコアでソートする。
- スコアは `-shanten*100000 + ukeire*200 + ukeireKinds*30 + han*80 + dora*60 - danger*weight` を基本とする。
- 相手リーチ時は、手牌の近さと危険度で `attack` / `balance` / `defense` を切り替える。
- 4人対戦の評価は `scripts/tournament.ts` で行う。勝率だけでなく、放銃率、平均順位、平均収支を見る。
- 推奨は `recommend()` がリーチ判定と攻守バイアスを付けて返す。
